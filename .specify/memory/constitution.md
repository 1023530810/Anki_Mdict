# ListenSpeak 项目章程

## 核心原则

### I. 代码质量

代码质量是项目可维护性和可扩展性的基础。所有代码必须遵循以下标准：

- **类型安全**: Python 代码必须使用类型注解；Anki 相关 import 必须置于 `TYPE_CHECKING` 下以支持静态分析
- **模块化设计**: 遵循分层架构（UI → 业务逻辑 → 媒体处理 → FFmpeg），各层职责清晰分离
- **命名规范**: 媒体文件命名 `ls_{source_hash}_{segment_index}.{ext}`，录音文件命名 `ls_rec_{note_id}_{timestamp}.{webm|m4a}`
- **代码审查**: 所有变更必须经过审查，确保符合项目架构和编码标准
- **无死代码**: 删除未使用的代码，不保留注释掉的代码块

**理由**: Anki 插件运行在用户本地环境，代码质量直接影响用户体验和问题排查效率。

### II. 测试标准

测试是保证功能正确性的关键手段。测试必须满足以下要求：

- **测试隔离**: 测试必须使用 `importlib.util` 直接加载模块，避免触发 `aqt` 依赖
- **测试覆盖**: 核心模块（media_processor、subtitle_parser、card_generator）必须有单元测试
- **测试命令**: 使用 `python -m pytest Anki_ListenSpeak/tests/ -v --tb=short` 运行全部测试
- **测试优先**: 新功能开发应先编写测试用例，确保测试失败后再实现功能
- **边界测试**: 必须覆盖字幕编码检测、FFmpeg 路径检测、多轨字幕对齐等边界场景

**理由**: 插件依赖外部工具（FFmpeg）和多种文件格式（SRT/VTT/ASS），测试是确保兼容性的唯一可靠方式。

### III. 用户体验一致性

用户体验必须在所有平台和场景下保持一致：

- **跨平台兼容**: 播放器和录音功能必须在桌面（Windows/macOS/Linux）和移动端（AnkiDroid/AnkiMobile）正常工作
- **格式适配**: 录音格式按平台自动选择（桌面/Android 用 WebM，iOS 用 M4A）
- **快捷键统一**: 空格播放/暂停，R 录音，[/] 调速，箭头键 seek/音量
- **错误提示**: 所有用户可见的错误信息必须使用中文，清晰说明问题和解决方案
- **UI 语言**: 默认语言为中文（注释、UI 字符串、文档）

**理由**: Anki 用户在多设备间同步学习，体验不一致会严重影响学习效率。

### IV. 性能要求

性能直接影响用户的学习体验，必须满足以下标准：

- **媒体处理**: 视频分段使用直接复制（无转码），音频转码为 WebM/Opus 以平衡质量和体积
- **响应时间**: 播放器控制（播放/暂停/调速）响应时间必须 < 100ms
- **内存占用**: 单张卡片的 JS 内存占用不得超过 50MB
- **文件大小**: 分段后的媒体文件应尽量小，避免同步时占用过多带宽
- **异步处理**: 导入工作流必须异步执行，不阻塞 Anki 主界面

**理由**: 用户可能导入大量媒体文件，性能问题会导致 Anki 卡顿甚至崩溃。

## 技术约束

### 依赖管理

- **FFmpeg**: 必需外部依赖，通过 `shutil.which()` 或用户配置路径检测
- **chardet**: 可选依赖，用于字幕编码自动检测
- **PyQt6/PyQt5**: 由 Anki 运行时提供，不单独打包

### 文件组织

```
Anki_ListenSpeak/
├── src/listen_speak/     # 插件源码
│   ├── templates/        # 卡片模板 HTML/CSS
│   └── media/            # 模板 JS 资产
└── tests/                # 单元与集成测试
```

### 模板架构

- JS 代码采用 IIFE 与命名空间模块（`LS.Player`、`LS.Recorder`、`LS.Shortcuts`）
- 模板通过 `pycmd()` 与插件通信，消息格式为 `ls_{action}:{payload}`

## 开发工作流程

### 代码提交

1. 所有变更必须在功能分支上开发
2. 提交前必须运行测试并确保全部通过
3. 提交信息使用中文，格式为 `[模块] 变更描述`

### 功能开发

1. 先在 `specs/` 目录创建功能规格文档
2. 编写测试用例并确认失败
3. 实现功能并确保测试通过
4. 更新相关文档

### 问题修复

1. 先编写复现问题的测试用例
2. 修复问题并确保测试通过
3. 验证修复不影响其他功能

## 治理

本章程是项目开发的最高指导原则，所有开发活动必须遵循章程规定。

- **章程优先**: 章程规定优先于其他实践和约定
- **修正流程**: 章程修正需要文档化变更理由、影响范围和迁移计划
- **合规验证**: 所有 PR 必须验证是否符合章程原则
- **复杂性证明**: 任何超出章程规定的复杂性必须提供充分理由
